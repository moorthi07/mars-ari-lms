Module 5 — Dataset Engineering

Dataset engineering is the most underrated skill in Physical AI. Most practitioners focus on models. The practitioners who get results focus on data. This module teaches you how to design, collect, store, and clean robot datasets.

The Dataset Schema

Every robot dataset frame contains three things:

Frame {
  observation: {
    image_wrist: [H, W, 3],      # wrist camera RGB
    image_overhead: [H, W, 3],   # overhead camera RGB (optional)
    joint_positions: [6],         # joint angles in radians
    joint_velocities: [6],        # joint speeds
    gripper_state: float,         # 0.0 = closed, 1.0 = open
  },
  action: {
    joint_delta: [6],             # change in joint positions
    gripper_action: float,        # -1 = close, +1 = open
  },
  timestamp: float,               # seconds since episode start
  episode_id: str,
  frame_index: int,
  task_description: str,          # "pick up red cube and place in bin"
  success: bool,                  # only set on final frame
}

Synchronization — The Silent Killer

Camera images and joint states are captured by different hardware at different rates. If they're not synchronized, your model sees a joint state that doesn't match the image. This is one of the most common causes of poor policy performance.

How to synchronize:

  • Use a single timestamp source (system clock or hardware trigger)
  • Record camera at fixed FPS (30 Hz typical)
  • Record joint states at same or higher rate
  • Interpolate joint states to match camera timestamps
  • Always log the timestamp with every data point

Data Cleaning

Raw teleoperation data is always noisy. Before training, you must clean it:

  1. Remove idle segments — Frames where the robot isn't moving (operator paused). Threshold: if joint velocity < 0.01 rad/s for >0.5s, mark as idle.
  2. Remove failed episodes — Episodes where the task wasn't completed. Keep only successes (or explicitly label failures).
  3. Clip extreme values — Joint velocities or forces outside physical limits indicate sensor errors.
  4. Normalize — Scale observations to [-1, 1] or [0, 1] range. Use dataset statistics (mean, std) stored in metadata.
  5. Visual inspection — Replay 10% of episodes and watch them. You'll catch issues no algorithm finds.

Distribution Shift

Distribution shift is when the data your policy was trained on doesn't match what it sees during deployment. Common causes:

  • All training episodes started from the same position → policy fails from new positions
  • Training data had consistent lighting → policy fails under different lighting
  • Operator always approached from the same angle → policy can't generalize

Fix: Deliberately vary your collection conditions. Different starting positions, lighting, object orientations. This is called coverage.
