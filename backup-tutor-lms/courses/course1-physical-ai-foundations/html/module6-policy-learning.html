Module 6 — Policy Learning

A policy is a function that maps observations to actions. Training a policy means teaching a neural network to reproduce the behavior in your dataset. This module covers the progression from a simple MLP baseline to vision-based policies.

What Is a Policy?

π(a | s) — given state s, output action a

In Physical AI, the policy is a neural network. It takes the robot's current observation (image + joint states) and outputs the next action (joint delta + gripper command).

Behavior Cloning — The Baseline

Behavior cloning (BC) is the simplest form of imitation learning. It treats the problem as supervised learning: given an observation, predict the action the human took. 

# Training loop (simplified)
for obs, action in dataset:
    predicted_action = policy(obs)
    loss = MSE(predicted_action, action)
    loss.backward()
    optimizer.step()

That's it. Behavior cloning is just supervised regression on (observation, action) pairs.

The Policy Learning Ladder

Start simple. Add complexity only when the simpler version fails.

Training Loop

  1. Load dataset — Split into train (80%) and validation (20%)
  2. Normalize — Scale observations and actions to [-1, 1]
  3. Forward pass — Run observation through policy network
  4. Compute loss — MSE between predicted and actual action
  5. Backward pass — Compute gradients
  6. Update weights — Adam optimizer, lr=1e-4
  7. Validate — Check loss on held-out episodes every N steps
  8. Evaluate in sim — Run rollouts, measure task success rate

Evaluation — The Only Metric That Matters

Validation loss tells you if the model is overfitting. It does NOT tell you if the robot can complete the task. The only real metric is task success rate: run 20 rollouts in simulation, count how many succeed.

A policy with low validation loss but 0% success rate is useless. A policy with higher loss but 80% success rate is what you want.

Common Failure Modes

  • Drift — Small errors compound over time. Policy reaches a state never seen in training. Fix: add more diverse demonstrations, use DAgger.
  • Oscillation — Policy can't commit to an action, oscillates between two options. Fix: smooth the action output, increase dataset size.
  • Mode collapse — Policy always does the same thing regardless of input. Fix: check dataset diversity, increase model capacity.
  • Overfitting — Perfect on training data, fails on new positions. Fix: more data, data augmentation, regularization.

DAgger — Dataset Aggregation

DAgger (Dataset Aggregation) fixes the drift problem. After training an initial policy:

  1. Run the policy in simulation
  2. When it drifts into a new state, record what a human would do from that state
  3. Add those new (state, action) pairs to the dataset
  4. Retrain
  5. Repeat

This is the same idea as Human-in-the-Loop (HIL) data collection from Module 4.
