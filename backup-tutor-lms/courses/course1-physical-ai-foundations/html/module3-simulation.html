Module 3 — Simulation Environments

Simulation is a data generator. Before you can train a robot policy, you need thousands of episodes. Real hardware is slow, expensive, and breaks. Simulation gives you unlimited, free, parallelizable data.

Why Simulate?

  • Safe — No hardware damage when the policy fails
  • Fast — Run 1000 episodes in parallel on a GPU
  • Cheap — No wear on motors, no broken grippers
  • Controllable — Change lighting, friction, object positions programmatically
  • Reproducible — Same seed = same episode every time

The Three Simulators You Need to Know

Recommendation: Start with MuJoCo. It runs in the browser (MuJoCo WASM), has excellent Python bindings, and is the standard for manipulation research. Move to Isaac Sim when you need photorealistic rendering for sim-to-real transfer.

Robot Models: URDF, MJCF, USD

A robot model is a file that describes the robot's physical structure:

  • URDF (Unified Robot Description Format) — XML, used by ROS and Gazebo. Describes links, joints, mass, collision geometry.
  • MJCF (MuJoCo XML) — MuJoCo's native format. More expressive for contact physics.
  • USD (Universal Scene Description) — NVIDIA's format for Isaac Sim. Supports photorealistic materials and complex scenes.

Most robot manufacturers provide URDF files. Tools like urdf2mjcf convert between formats.

The Simulation Control Loop

while not done:
    obs = env.get_observation()    # camera image + joint states
    action = policy(obs)           # your model or teleop input
    env.step(action)               # physics advances one timestep
    reward = env.get_reward()      # optional, for RL
    data.record(obs, action)       # save to dataset

This loop is identical whether you're collecting teleop data, running a trained policy, or doing RL. The only thing that changes is where action comes from.

Reset and Determinism

Every training episode starts with a reset — the environment returns to a known starting state. This is critical for:

  • Reproducibility (same seed = same episode)
  • Curriculum learning (start easy, gradually increase difficulty)
  • Domain randomization (randomize object positions, lighting, friction on each reset)

Always set a random seed when collecting data. This lets you reproduce any episode for debugging.

Common Simulation Mistakes

  • ❌ No noise — Real sensors are noisy. Add Gaussian noise to joint readings and camera images.
  • ❌ Perfect friction — Real surfaces slip. Randomize friction coefficients.
  • ❌ No latency — Real control loops have delays. Add action delay of 1-3 timesteps.
  • ❌ Unrealistic lighting — Real cameras see shadows and glare. Vary lighting in sim.

These mistakes cause the sim-to-real gap — policies that work perfectly in sim but fail on real hardware. We cover this in Module 8.
