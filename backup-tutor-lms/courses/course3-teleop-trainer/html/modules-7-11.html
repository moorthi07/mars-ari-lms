Module 7 — Data Labeling

Labels are the supervision signal that tells the AI what each part of your data means. Good labels enable better generalization. This module covers what to label and how.

What Labels Does a VLA Need?

For a VLA model, the primary labels are already in your dataset — the actions you performed. But additional labels significantly improve training:

  • Task description — Natural language description of the full task (from Module 5)
  • Phase labels — Which phase of the task is this frame? (approach, grasp, lift, place)
  • Object labels — What objects are in the scene? Where are they?
  • Success/failure — Did this episode succeed?
  • Key frames — Which frames are most important? (moment of grasp, moment of release)

Phase Labeling

Phase labels divide an episode into meaningful segments. This helps the model understand task structure — not just "what action to take" but "what stage of the task am I in?"

Episode phases for "pick and place":
  t=0.0 to t=1.2  → phase: "approach"
  t=1.2 to t=1.8  → phase: "grasp"
  t=1.8 to t=2.5  → phase: "lift"
  t=2.5 to t=3.8  → phase: "transport"
  t=3.8 to t=4.5  → phase: "place"
  t=4.5 to t=5.0  → phase: "retract"

Object Annotation

For vision-based policies, knowing where objects are in the image helps the model learn faster. Basic object annotation includes:

  • Bounding boxes around target objects
  • Object class labels ("red cube", "blue bin")
  • Object state ("grasped", "released", "in-bin")

Tools: Label Studio (open source), Roboflow (free tier), or the platform's built-in labeling interface.

Key Frame Selection

Not all frames are equally important. Key frames are the moments that define the task:

  • The frame just before grasping (gripper approaching object)
  • The frame of successful grasp (gripper closed on object)
  • The frame of placement (object at target location)

Some training methods (like ACT) use key frames to define action chunks — sequences of actions that lead to the next key frame.

Labeling Efficiency Tips

  • Label in batches — do all episodes of one task type together
  • Use keyboard shortcuts in labeling tools
  • Label phase boundaries first (fast), then object annotations (slower)
  • Have a second person review 10% of labels for consistency

Module 8 — Simulation for Data Generation

Real robot data is expensive and slow to collect. Simulation lets you generate thousands of episodes automatically. This module covers when and how to use simulation to augment your real data.

When to Use Simulation

Use Simulation When...Use Real Data When...
You need 1000+ episodesYou need <200 episodes
Task is well-defined and repeatableTask requires real-world nuance
You want to test policy variations quicklyYou're doing final validation
Hardware is unavailable or expensiveSim-to-real gap is critical
You want domain randomizationYou need exact real-world physics

MuJoCo — Free, Browser-Based

MuJoCo runs in your browser (no GPU needed). Best for:

  • Quick prototyping and testing
  • Manipulation tasks with simple physics
  • Learning and experimentation

Access it directly in the MarsGeo platform — no setup required.

Genesis & Gazebo — Cloud CPU (Starter+)

More capable simulators that run on cloud CPU instances. Best for:

  • More complex physics (fluid, deformable objects)
  • Mobile robot navigation
  • Outdoor environments

Requires a Starter subscription. Billed per hour of compute time.

Isaac Sim — Cloud GPU (Pro+)

NVIDIA's photorealistic simulator. Best for:

  • Sim-to-real transfer (photorealistic rendering reduces the gap)
  • Large-scale data generation (run 100 parallel environments)
  • Humanoid and complex robot simulation

Requires a Pro subscription. Significantly more expensive but produces the best sim-to-real results.

Domain Randomization in Practice

When generating sim data, always enable domain randomization:

  • Randomize object position: ±5cm from nominal
  • Randomize object color/texture: 5-10 variations
  • Randomize lighting: 3-5 lighting configurations
  • Randomize camera position: ±2cm, ±5°

This makes your sim data more diverse and reduces the sim-to-real gap when you deploy to real hardware.

Module 9 — Configuring VLA & VLM Models

As a trainer, you don't build models from scratch — you configure and feed them. This module explains what you control and what you don't, and how to set up a model training run.

What a Trainer Controls

You are not a model researcher. Your job is to:

  • Prepare the dataset (Modules 2-7)
  • Select the right model architecture for your task
  • Configure training hyperparameters
  • Monitor training progress
  • Evaluate the trained policy
  • Decide when to collect more data

Choosing a Model

ModelBest ForData NeededTraining Time
ACT (Action Chunking Transformer)Dexterous manipulation, fast tasks50-200 episodes1-4 hours (GPU)
Diffusion PolicyMulti-modal tasks, complex trajectories100-500 episodes4-12 hours (GPU)
SmolVLALanguage-conditioned, multi-task500+ episodes8-24 hours (GPU)
OpenVLA (fine-tune)Language-conditioned, strong baseline200+ episodes4-8 hours (GPU)

Key Training Hyperparameters

These are the parameters you'll typically configure:

  • Batch size — How many episodes to process at once. Larger = faster but needs more GPU memory. Start with 32.
  • Learning rate — How fast the model updates. Too high = unstable training. Too low = slow convergence. Start with 1e-4.
  • Training steps — How long to train. More steps = better performance up to a point. Monitor validation loss to decide when to stop.
  • Chunk size — For ACT: how many future actions to predict at once. Typical: 50-100 frames.

Monitoring Training

Watch these metrics during training:

  • Training loss — Should decrease steadily. If it plateaus early, you may need more data.
  • Validation loss — Should decrease with training loss. If it increases while training loss decreases, you're overfitting.
  • Success rate in sim — Run evaluation rollouts every N steps. This is the real metric.

When to Stop Training

  • Validation loss has plateaued for 10+ epochs
  • Success rate in sim has plateaued
  • Success rate in sim is >80% (good enough to deploy and collect HIL data)

Module 10 — Managing Data Collection Jobs

As a trainer, you're often the client on TeleOp Hub — posting jobs for controllers to complete, reviewing their work, and managing the data pipeline.

Writing a Good Job Description

A clear job description produces better data. Include:

  • Task description — Exactly what the robot should do, step by step
  • Robot type — Which robot, which controls to use
  • Episode requirements — How many episodes, how long each should be
  • Quality criteria — What makes a good episode (smooth movement, specific approach angle, etc.)
  • Reset procedure — Exactly how to reset between episodes
  • What to avoid — Common mistakes to prevent

Example Job Description

Task: Pick up the red cube from the table and place it in the blue bin.

Robot: 6-DOF arm with wrist camera. Use arm joystick controls.

Episodes needed: 100 successful episodes

Episode length: 5-10 seconds

Reset: Return arm to home position. Place red cube at marked position (center of table, 30cm from arm base).

Quality criteria: Smooth approach from above. Gripper fully closed before lifting. No pauses >0.5s.

Avoid: Approaching from the side. Dropping the cube. Idle time.

Reviewing Submitted Work

When controllers submit sessions, review them before accepting:

  1. Watch 10% of episodes in the media gallery
  2. Check telemetry charts for idle time and jitter
  3. Verify task completion rate (>90% target)
  4. Check episode length consistency
  5. Accept or request revisions

Quality Gates

Set minimum quality thresholds before accepting a batch:

  • Task success rate: >90%
  • Idle ratio: <5% of frames
  • Episode length: within ±50% of target
  • Video quality: no blurry or dark episodes

Module 11 — The Iteration Loop

Physical AI is not a one-shot process. It's an iterative loop of collect, train, evaluate, and improve. This final module teaches you how to run that loop effectively.

The Full Loop

1. Design session (Module 2)
2. Collect data (Modules 3-5)
3. Curate data (Module 6)
4. Label data (Module 7)
5. Train policy (Module 9)
6. Evaluate in simulation
7. Identify failure modes
8. Design targeted collection to fix failures
9. Collect more data (targeting failures)
10. Retrain with augmented dataset
11. Deploy to real hardware (with safety layer)
12. Collect HIL data (human corrects real failures)
13. Fine-tune on HIL data
14. Go to step 6

Reading Evaluation Results

After training, run 20 evaluation rollouts in simulation. For each failure, ask:

  • Where did it fail? — Which phase of the task? (approach, grasp, place?)
  • Why did it fail? — Wrong position? Wrong timing? Didn't see the object?
  • What data would fix it? — More episodes from that starting position? More variation in object placement?

Common Failure Patterns and Fixes

Failure PatternRoot CauseFix
Fails from new starting positionsNot enough position coverageCollect from 5+ different starting positions
Grasps but drops objectInconsistent grasp in training dataCollect 50 episodes focused on grasp quality
Works in sim, fails on real hardwareSim-to-real gapCollect real data, add domain randomization to sim
Oscillates near targetNoisy training data near completionCurate episodes more carefully, remove jitter
Ignores language instructionInsufficient language diversityCollect data with more varied task descriptions

When You're Done

A policy is "done" when it meets your performance target consistently. Typical targets:

  • Simple pick-and-place: 90%+ success rate in sim, 80%+ on real hardware
  • Multi-step manipulation: 80%+ in sim, 70%+ on real hardware
  • Language-conditioned: 70%+ on seen tasks, 50%+ on unseen tasks
